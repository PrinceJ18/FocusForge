# supabase-rules.md

# FocusForge Supabase Standards & Database Rules

Version: 1.0

---

# Purpose

This document defines how every AI agent must interact with Supabase.

Supabase is the authoritative backend for this application.

Every database change must prioritize:

* Security
* Correctness
* Performance
* Maintainability
* Scalability

Never bypass these principles.

---

# Backend Philosophy

Supabase is responsible for:

Authentication

Authorization

Database

Storage

Security Policies

Future Realtime Features

Business logic should remain predictable and secure.

---

# Source of Truth

Supabase is the only source of truth.

Never duplicate database state unnecessarily.

Avoid maintaining multiple conflicting copies of the same data.

---

# Authentication Rules

Authentication is centralized.

Never create custom authentication systems.

Always use Supabase Auth.

Supported authentication:

* Email & Password
* Google OAuth

Future providers should integrate into the same authentication flow.

---

# Session Management

Always verify:

Authenticated user

Session validity

Token expiration

User ID

Never assume the user is authenticated.

Always handle unauthenticated states gracefully.

---

# Authorization

Authentication answers:

"Who is the user?"

Authorization answers:

"What may this user access?"

Never confuse the two.

Authorization must always be enforced using Row Level Security.

---

# Row Level Security (RLS)

Every user-owned table must have RLS enabled.

Policies must prevent users from accessing other users' data.

Never disable RLS for convenience.

Never rely only on frontend filtering.

Database security is mandatory.

---

# Database Design

Each table should have one responsibility.

Examples:

profiles

tasks

expenses

focus_sessions

budgets

savings_goals

categories

rewards

Do not mix unrelated entities into one table.

---

# Primary Keys

Every table must use a stable primary key.

Avoid using mutable fields as identifiers.

Prefer UUIDs generated by Supabase.

---

# Foreign Keys

Use foreign keys wherever relationships exist.

Examples:

Expense → User

Task → User

Goal → User

Never duplicate relationship data.

---

# Naming Conventions

Tables

snake_case

Examples:

focus_sessions

savings_goals

custom_categories

Columns

snake_case

Examples:

created_at

updated_at

user_id

expense_amount

Never mix naming conventions.

---

# Timestamps

Every important table should include:

created_at

updated_at

Use database timestamps whenever possible.

---

# User Ownership

Every user-owned record should contain:

user_id

Every query should respect ownership.

Never expose another user's data.

---

# Query Design

Queries should be:

Small

Readable

Efficient

Predictable

Request only the required columns.

Avoid SELECT * unless justified.

---

# Filtering

Always filter as early as possible.

Example:

Current user

↓

Date

↓

Category

↓

Limit

Avoid fetching unnecessary rows.

---

# Pagination

Large datasets should support pagination.

Avoid loading entire tables into memory.

Support future scalability.

---

# Indexes

Indexes should exist for frequently queried columns.

Examples:

user_id

created_at

category_id

status

Avoid unnecessary indexes.

---

# Transactions

When multiple related operations must succeed together,

prefer transactional behavior where supported.

Avoid partial updates.

Maintain data consistency.

---

# Validation

Validate data before writing.

Never trust client input.

Validate:

Required fields

Numeric ranges

Dates

Enums

Relationships

Prevent invalid data from reaching the database.

---

# Error Handling

Always handle:

Connection failures

Permission failures

Timeouts

Missing data

Duplicate records

Unexpected responses

Provide meaningful user feedback.

---

# Storage Rules

Store only necessary files.

Validate file types.

Validate file size.

Avoid exposing private files publicly.

Use secure storage paths.

---

# Environment Variables

Never hardcode:

Project URL

Anon Key

Service Role Key

Secrets

Use environment variables only.

Never commit secrets to Git.

---

# Service Role Key

The Service Role Key must never be exposed to the frontend.

Never include it in:

React

TypeScript

Client components

Public repositories

Treat it as confidential.

---

# Realtime

Realtime features should:

Minimize unnecessary subscriptions.

Unsubscribe correctly.

Avoid duplicate listeners.

Handle reconnects gracefully.

---

# Database Migrations

Schema changes should occur through migrations.

Avoid manually editing production databases.

Every migration should be:

Repeatable

Documented

Reversible where practical

---

# Backup Philosophy

Never assume data can be recreated.

Design features with data integrity in mind.

Protect user information.

---

# Performance

Avoid:

Repeated queries

Duplicate requests

N+1 query patterns

Fetching unused columns

Large payloads

Optimize only after measuring.

---

# Security Checklist

Before shipping any database feature verify:

✓ RLS enabled

✓ Queries filtered

✓ User ownership respected

✓ Validation present

✓ Errors handled

✓ Secrets protected

✓ Environment variables used

✓ Service role key hidden

✓ No sensitive logs

---

# AI Database Workflow

Whenever modifying the backend:

Read schema

Understand relationships

Review RLS

Review existing queries

Review types

Implement

Test

Review

Document

Never modify the schema without understanding its impact.

---

# Anti-Patterns

Never:

Disable RLS

Hardcode keys

Trust frontend validation

Duplicate user data

Fetch unnecessary data

Ignore errors

Store sensitive data insecurely

Expose service role keys

---

# Final Rule

Supabase is the security boundary of FocusForge.

Every backend change must improve:

Security

Reliability

Performance

Maintainability

Scalability

If uncertain, choose the safer implementation over the faster one.
